# SpecFlow 用户可见话术规范（单一事实来源）

本目录下所有 **orchestration-*.md**、**completion-output-*.md**，以及 **`tools/user-facing.cjs`** 里生成给 **终端用户 / 业务方** 的文案，均须遵守本节。目标是：**像流程助手一样**，用**自然、简短**的中文推进，**不暴露**内部实现与机读字段。

---

## 1. 语气与结构

- **人称**：面向用户说明「你 / 请」，避免「用户」「终端」等元称呼。
- **长度**：标题一行、正文若干短句；能一句说清不写两段。
- **确定性**：用「先…再…」「选一项即可」；少用被动与从句套娃。

---

## 2. 禁止出现在用户可见文案中的内容（除非用户主动调试）

以下内容**仅**允许出现在代理/编排技能、脚本注释或故障排查文档中，**不得**作为对用户展示的主文案：

| 类别 | 示例（禁止） |
|------|----------------|
| 内部标识 | `CQ-xx`、`[?]`、`Section 5`、Clarification Log、锚点、`<!-- specflow:... -->` |
| 脚本/引擎 | `orchestrator.cjs`、`specflow-engine`、`dispatch`、`suggestedAction`、`gates.`、`pending-protocol.json` |
| 子代理/技能名 | `specflow-specify`、`print-protocol`、MUST/STRICTLY PROHIBITED 对用户复述 |
| 路径与仓库结构 | `ai-docs/`、`specify.md`（可对用户说「需求说明文档」「规格文档」） |
| 字段名 | `requirementId`、`interaction_required`、`init_requirement_id` |
| 对用户的「解释原因」里夹带运行机制 | 「按引擎约定」「门禁」「再跑引擎」「点确认后执行 xxx 脚本」等 |

**例外**：需求编号本身可以说「当前需求编号」「编号 xxx」。

**例外（SNR 动作优先）**：编排助手展示「下一小步」说明时，**第一行**固定为英文 `Next Actions (Remaining: N)`（`N` 为结构化残差合计 **totalScore**：未闭环验收项 + 最近一次检查未通过项 + 待关闭门禁 + 待补验收证据任务），其后仍为中文模板正文；该行用于度量与动作优先，不视为「暴露内部字段」。

### 2.1 反机械化句式（新增）

以下句式属于“执行日志口吻”，会增加理解成本，**禁止**直接对用户说：

- 「按 implement 链执行」
- 「拿到引擎返回的可执行任务」
- 「实现入口脚本路径和当前仓库不一致，我先定位 orchestrator.cjs / specflow-engine.cjs」
- 「被卡在一个交互点」
- 「继续跑同一条流程」

改写原则（固定顺序）：

1. 先说**当前进展结论**（发生了什么）
2. 再说**用户此刻只需做的一件事**（选项/确认/补一句）
3. 最后说**完成后会继续推进**（不解释内部机制）

推荐改写示例：

- 「我先推进到下一步，拿到明确待办后立刻开始实现。」
- 「这里需要你先选一个选项，我收到后马上继续。」
- 「这个问题定下来后，我会直接往下执行并同步结果。」

### 2.2 子阶段结束汇报、主对话总结（与 `completion-output-*.md` 对齐）

凡面向用户/业务方的**结论、复盘、下一步**，包括子代理完成后的汇报，**必须**只写：业务进展、待决策点、一句下一步（见各 `completion-output-*.md`）。**禁止**在同一轮对用户说明「如何驱动流程」（那是编排侧内部操作）；**禁止**列出「你可直接打开的文件：路径…」或「我会再执行 xxx 再 yyy」。

---

## 3. 推荐用语（阶段与动作）

对用户说明「在干什么」时，优先用左列，避免右列（内部说法）：

| 对用户 | 内部/文档可能写法 |
|--------|---------------------|
| 需求说明 / 业务口径 | Specify、specify.md |
| 技术方案 / 接口与任务拆分 | Plan、plan.md |
| 开发与自测 | Implement、Roadmap |
| 验收 / 检查 | QA |
| 收尾归档 | Archive |
| 待确认的事 / 要先定下来的选项 | 澄清、CQ、未闭合项 |
| 点选 / 在对话里说一句 | AskQuestion、interaction |
| 业务知识库（当前需求业务规则活文档） | 需求目录下 `business-domains/`（对用户不说路径名，用左列） |

---

## 3.1 知识闭环推荐话术（新增）

- **初始化**：`正在通过骨架探测初始化项目知识库...`
- **开发中**：`检测到业务逻辑增量，已准备好在归档时同步至全局资产...`
- **完成时**：`当前实现已通过项目 code-style 规范对齐检查。`

## 3.2 提效与自动化推荐话术（EBA）

- **无感澄清**：`根据领域知识库，我已自动解决了若干处待确认项，请在需求说明里快速审阅后继续。`
- **并行推进**：`正在并行推进多个相互独立的任务组（不影响彼此）；任一任务组完成后会立即进入该组下一步。`
- **自愈重试**：`检测到验收未通过，正在尝试自动修复并重新验证...`

以上句式可按上下文微调，但要保持同一语义，不要暴露脚本名与内部字段。

---

## 4. 修改流程（避免「改一处忘一处」）

### 4.0 模板增长控制（新增）

- 默认走统一自然语言渲染（结论 → 用户动作 → 我会继续推进），不要为每个内部状态单独新增模板文件。
- 仅以下高风险场景允许专用模板：`anchor`、`block`、`requirement_id.conflict`。
- 新增模板前先判断：是否涉及不可歧义确认或风险提示；否则应复用通用渲染。

1. **先**对照本文件：新文案是否违反第 2 节、是否符合第 3 节。
2. 再改具体文件：
   - 编排入口、澄清摘要 → `user-facing.cjs` + `orchestration-*.md`
   - 子阶段结束汇报 → `completion-output-*.md`
   - 引擎弹窗内嵌长句（若仍有）→ `specflow-engine.cjs` 中对应 `prompt`/`reason`（须再用人话过一遍）
3. **不要**在 `orchestration-*.md` 里写「仅供代理」的长技术段；代理约束请放在 **`orchestrating-specflow`** 等技能中。

---

## 5. 自检清单（发版/改文案前扫一眼）

- [ ] 用户扫一眼能否理解「下一步要做什么」？
- [ ] 是否出现英文文件名、`.cjs`、驼峰字段名？
- [ ] 是否假设用户知道仓库目录或 Markdown 结构？
- [ ] 同一含义在全库是否用词一致（见第 3 节）？
- [ ] 子阶段汇报/总结是否**没有**路径、脚本名、引擎/弹窗字段名、运行机制（见第 2.1 节）？

---

## 6. 与引擎 JSON 的关系

引擎仍输出 `questions[].id` 等机读字段供工具使用；**展示给用户的 `prompt`/`headline`/`body`** 必须经过本规范过滤。`user-facing.cjs` 负责把 `block`/`interaction`/`dispatch` 映射为模板变量，**优先在这里集中改用户可见句**，避免在多处模板重复矛盾。
